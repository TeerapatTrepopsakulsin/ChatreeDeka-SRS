\subsection{Domain Model}
\label{section:domain-model}
\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{images/domain_model.png}
    \caption{ChatreeDeka Domain Model}
\end{figure}

ChatreeDeka's domain model is built around the CaseSimulation, which controls the overall flow of the legal dispute simulation. A User starts the process by entering the case facts, evidence, and legal claims. The system then coordinates several Agents to play different courtroom roles, such as the plaintiff, defendant, auditor, and judge. These agents interact by creating and reviewing Arguments, which can be supported by LegalProvisions and Precedents. During the simulation, the auditor checks the reliability of the arguments and records the results in AuditLogs, while the judge reviews the discussion and gives a Verdict. After the simulation is completed, the system generates a PostTrialAnalysis to explain the result, provide feedback, and suggest possible next steps.
\subsection{Architecture Design}
\label{section:architecture-design}
The software architecture follows a layered design, separating the presentation layer (user inputs and case visualization), the application layer (case orchestration and agent coordination), the AI/agent layer (role-specific reasoning and argument generation), and the data/knowledge layer (legal provisions, precedents, and audit logs). This separation keeps responsibilities clear, supports independent evolution of the AI components, and reduces coupling between the user interface and the underlying legal reasoning pipeline.

\subsection{System Overview}

\textbf{ChatreeDeka} is an AI-driven platform for simulating trials. It features \textbf{4 core agents} trained via Multi-Agent Reinforcement Learning (MARL), with \textbf{hybrid user control}:

\begin{itemize}[leftmargin=*]
\item \textbf{Judge Agent}: Manual user input (rulings, objections, procedure).
\item \textbf{Defender Lawyer Agent}: Auto-generates defenses, cross-exams (MARL-trained for realism).
\item \textbf{Prosecutor Lawyer Agent}: Manual user input (case-building, exams).
\item \textbf{Case Predictor Agent}: Analyzes full sim outputs (texts, evidence, results) to predict winning probability (\textbf{\%} for each side).
\end{itemize}

\subsection{Core Workflow}

\begin{enumerate}
\item Users input case details/evidence $\rightarrow$ simulation launches.
\item \textbf{Trial}: User manually provides Judge/Prosecutor lines; Defender auto-responds in real-time mock trial.
\item Users add conversation/evidence dynamically (e.g., ``only auto Defendant, manual others'', or ``all auto'').
\item \textbf{Post-trial} (user says ``end''), Case Predictor outputs win probability + feedback/pleadings/replay.
\end{enumerate}

\subsection{Evaluation}
% <TIP: Describe the metrics you plan to use to evaluate your AI model. />

Evaluation spans two dimensions: the accuracy of the judge agent's rulings and the behavioural quality of the multi-agent simulator. Each metric targets a distinct failure mode and is defined independently so that components can be assessed and iterated separately.

\subsubsection*{Judge Agent: Prediction Accuracy}

The judge agent is evaluated against labelled case outcomes from historical Thai civil court records. Accuracy measures the fraction of cases in which the agent's ruling matches the ground-truth disposition:

\[
\text{Accuracy} = \frac{\text{Correct Predictions}}{\text{Total Cases}}
\]

This metric is a necessary but not sufficient condition for deployment readiness. A high accuracy score without adequate procedural compliance (see below) would indicate a system that reaches correct conclusions through legally unsound reasoning paths.

\subsubsection{Simulator: Win Rate}

Win rate quantifies the strategic effectiveness of each lawyer agent across the full case set. Settlements are credited at half-weight to reflect partial success:

\[
\text{Win Rate} = \frac{\text{Wins} + 0.5 \times \text{Settlements}}{\text{Total Cases}}
\]

This metric provides a coarse signal for whether MARL training is converging toward competent adversarial behaviour. It is interpreted alongside the two quality metrics below, since a high win rate achieved through procedurally invalid arguments is not considered acceptable performance.

\subsubsection{Simulator: Procedural Compliance}

Procedural compliance measures how consistently agents operate within valid Thai civil procedure. An action is classified as valid if it is permissible at the current stage of the trial under the Civil Procedure Code:

\[
\text{Procedural Compliance} = \frac{\text{Valid Actions}}{\text{Total Actions}}
\]

Low compliance indicates that agents are exploiting simulation gaps rather than learning transferable courtroom behaviour, which directly undermines the training value of the platform.

\subsubsection{Simulator: Hallucination Rate}

Legal reasoning quality is assessed by auditing the statutory and precedent citations made by each agent. A claim is flagged as a hallucination when it cannot be traced to an extant provision of the Thai Civil and Commercial Code or to a citable court decision:

\[
\text{Hallucination Rate} = \frac{\text{Unverifiable Legal Claims}}{\text{Total Legal Claims}}
\]

Reducing hallucination rate is a prerequisite for the platform to serve as a meaningful legal training tool. Unverifiable citations would mislead users and erode the simulation's value as a proxy for real proceedings.

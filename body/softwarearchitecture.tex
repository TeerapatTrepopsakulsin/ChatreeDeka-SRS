ChatreeDeka's domain model is built around the CaseSimulation, which controls the overall flow of the legal dispute simulation. A User starts the process by entering the case facts, evidence, and legal claims. The system then coordinates several Agents to play different courtroom roles, such as the plaintiff, defendant, auditor, and judge. These agents interact by creating and reviewing Arguments, which can be supported by LegalProvisions and Precedents. During the simulation, the auditor checks the reliability of the arguments and records the results in AuditLogs, while the judge reviews the discussion and gives a Verdict. After the simulation is completed, the system generates a PostTrialAnalysis to explain the result, provide feedback, and suggest possible next steps.
\subsection{Architecture Design}
\label{section:architecture-design}
The software architecture follows a layered design, separating the presentation layer (user inputs and case visualization), the application layer (case orchestration and agent coordination), the AI/agent layer (role-specific reasoning and argument generation), and the data/knowledge layer (legal provisions, precedents, and audit logs). This separation keeps responsibilities clear, supports independent evolution of the AI components, and reduces coupling between the user interface and the underlying legal reasoning pipeline which shown in the Figure 4.2.

\subsection{System Overview}

\textbf{ChatreeDeka} is an AI-driven platform for simulating trials. It features \textbf{4 core agents} trained via Multi-Agent Reinforcement Learning (MARL), with \textbf{hybrid user control}:

\begin{itemize}[leftmargin=*]
\item \textbf{Judge Agent}: Manual user input (rulings, objections, procedure).
\item \textbf{Defender Lawyer Agent}: Auto-generates defenses, cross-exams (MARL-trained for realism).
\item \textbf{Prosecutor Lawyer Agent}: Manual user input (case-building, exams).
\item \textbf{Case Predictor Agent}: Analyzes full sim outputs (texts, evidence, results) to predict winning probability (\textbf{\%} for each side).
\end{itemize}

\subsection{Core Workflow}

\begin{enumerate}
\item Users input case details/evidence $\rightarrow$ simulation launches.
\item \textbf{Trial}: User manually provides Judge/Prosecutor lines; Defender auto-responds in real-time mock trial.
\item Users add conversation/evidence dynamically (e.g., ``only auto Defendant, manual others'', or ``all auto'').
\item \textbf{Post-trial} (user says ``end''), Case Predictor outputs win probability + feedback/pleadings/replay.
\end{enumerate}

\subsection{Data Sources and Collection}

To ensure the Multi-Agent AI system generates legally accurate and verifiable arguments, the ChatreeDeka platform relies on a highly structured Retrieval-Augmented Generation (RAG) database. The data collection and preparation pipeline is divided into the following phases:
\textbf{1. Primary Data Sources and Scope}
The foundational dataset for this system consists of statutory laws and legal precedents. The primary data is sourced from public domain government databases, specifically the official publications by the Office of the Council of State of Thailand. 
\begin{itemize}
    % TODO 
    \item \textbf{Statutory Law:} The complete text of the Thai Civil and Commercial Code (TCCC), encompassing all sections (e.g., Book I: General Principles, Book II: Obligations, etc.).
    \item \textbf{Case Law (Deka):} A curated subset of historical Thai Supreme Court rulings relevant to the targeted civil disputes (e.g., breach of contract, torts, and family law).
\end{itemize}

\textbf{2. Anticipated Challenges}
Handling Thai legal text presents significant Natural Language Processing (NLP) challenges. Unlike English, the Thai language does not use spaces to separate words. Furthermore, Thai legal documents utilize highly formal, archaic, and context-dependent terminology. This creates a dual challenge: accurate word tokenization during the data cleaning phase and ensuring the vector embedding model correctly captures the semantic nuance of legal Thai rather than translating it literally.

\textbf{3. Data Preparation and Cleaning}
Before the data can be utilized by the agents, it must be transformed from raw text into a machine-readable vector format. The preparation pipeline involves the following steps:
\begin{itemize}
    \item \textbf{Extraction and Normalization:} Raw data (often in PDF or HTML formats) is extracted and stripped of formatting artifacts. Standardized headers, footers, and page numbers are removed to prevent noise in the RAG retrieval process.
    \item \textbf{Semantic Chunking:} Instead of arbitrarily splitting the text by character count, the data is strictly chunked by legal section (e.g., Section 15, Section 19). This ensures that each chunk contains a complete, isolated legal rule, preventing the LLM from conflating overlapping statutes.
    \item \textbf{Vector Embedding:} The cleaned, sectioned text is processed through a multilingual or Thai-optimized embedding model (such as BGE-m3). The resulting high-dimensional vectors are stored in a Vector Database (e.g., ChromaDB or Pinecone). When a user inputs a case fact, the system queries this database to retrieve the exact TCCC sections required for the agents to construct their debate.
\end{itemize}
\subsection{Model Design}
\label{subsection:model-design}

The ChatreeDeka platform employs a Heterogeneous Multi-Agent Architecture, utilizing specifically chosen Large Language Models (LLMs) for distinct courtroom roles. This design is strictly constrained by local hardware limitations (a 12GB VRAM limit), necessitating highly optimized, quantized models that load sequentially in a synchronous ``fail-fast'' pipeline. All models are prompted in Thai and constrained to output arguments based exclusively on the Retrieval-Augmented Generation (RAG) database of the Thai Civil and Commercial Code (TCCC).

\subsubsection{1. The Judge: DeepSeek-R1-8B}
\noindent\textbf{Model Choice:} DeepSeek-R1 (8 Billion Parameters). \\
\textbf{Justification:} While standard LLMs output text linearly, legal adjudication requires deep logical deliberation. DeepSeek-R1 was selected specifically for its natively trained Chain-of-Thought (CoT) capabilities. Before rendering a final decision, the model utilizes an internal \texttt{<think>} token phase to weigh the plaintiff's and defender's arguments against the TCCC. \\
\textbf{Integration \& Prompt Design:} The Judge is prompted to act as a neutral Thai civil court magistrate. The prompt forces the model to read the verified arguments and output its step-by-step logical reasoning (visible to law students for educational purposes) before concluding with a formal judicial verdict and cost allocation. 

\subsubsection{2. The Lawyer (Plaintiff): Llama-3.2-3B-Instruct}
\noindent\textbf{Model Choice:} Meta's Llama-3.2 (3 Billion Parameters, Instruct-Tuned). \\
\textbf{Justification:} As discussed in the \textit{AgentsCourt} literature review, simulating a trial requires autonomous agents to generate procedural text. Llama-3.2-3B was chosen because its highly efficient parameter size allows it to run simultaneously with the Defender agent in VRAM. Despite its small size, its updated architecture handles multilingual context (Thai) exceptionally well when provided with strict RAG context. \\
\textbf{Integration \& Prompt Design:} The Plaintiff's system prompt restricts it from utilizing its pre-trained general knowledge. It is instructed to adopt an aggressive but legally sound posture, formulating its initial claims and demands based purely on the user's inputted facts and the retrieved TCCC sections.

\subsubsection{3. The Defender: Llama-3.2-3B-Instruct}
\noindent\textbf{Model Choice:} Meta's Llama-3.2 (3 Billion Parameters, Instruct-Tuned). \\
\textbf{Justification:} To ensure a fair and mathematically balanced simulation, the Defender utilizes the exact same foundational model as the Plaintiff. This creates ``Symmetrical Warfare,'' ensuring that neither side has an inherent computational advantage in reasoning depth. \\
\textbf{Integration \& Prompt Design:} The Defender's prompt is designed adversarially. Upon receiving the Plaintiff's generated argument, the Defender is instructed to identify statutory exceptions, mitigate damages, and construct counter-arguments using the same RAG-retrieved TCCC database. This dynamic push-and-pull is designed to challenge the law student using the platform.

\subsubsection{4. The Case Predictor (Auditor/Verifier): Qwen-2.5-7B-Instruct}
\noindent\textbf{Model Choice:} Alibaba's Qwen-2.5 (7 Billion Parameters, Instruct-Tuned). \\
\textbf{Justification:} Inspired directly by the \textit{Debate-Feedback} framework (Chen et al., 2025) discussed in Chapter 2, the system requires an independent Assistant/Verifier to evaluate the reliability of the agents' debate and prevent hallucinations. Qwen-2.5-7B is currently the state-of-the-art model in its size class for strict instruction-following, logical auditing, and precise JSON data extraction. \\
\textbf{Integration \& Prompt Design:} This model operates as a background checkpoint and does not ``roleplay.'' Its prompt is strictly analytical: \textit{``You are a Legal Auditor. Read the generated argument and the retrieved TCCC statutes. Does the argument hallucinate or misrepresent the law? Output PASS or FAIL, followed by a one-sentence reason.''} If Qwen outputs FAIL, the system intercepts the argument and forces the Lawyer or Defender to regenerate, ensuring that the Judge only predicts case outcomes based on 100\% verified Thai law.

\subsection{Evaluation}
% <TIP: Describe the metrics you plan to use to evaluate your AI model. />

Evaluation spans two dimensions: the accuracy of the judge agent's rulings and the behavioural quality of the multi-agent simulator. Each metric targets a distinct failure mode and is defined independently so that components can be assessed and iterated separately.

\subsubsection*{Judge Agent: Prediction Accuracy}

The judge agent is evaluated against labelled case outcomes from historical Thai civil court records. Accuracy measures the fraction of cases in which the agent's ruling matches the ground-truth disposition:

\[
\text{Accuracy} = \frac{\text{Correct Predictions}}{\text{Total Cases}}
\]

This metric is a necessary but not sufficient condition for deployment readiness. A high accuracy score without adequate procedural compliance (see below) would indicate a system that reaches correct conclusions through legally unsound reasoning paths.

\subsubsection{Simulator: Win Rate}

Win rate quantifies the strategic effectiveness of each lawyer agent across the full case set. Settlements are credited at half-weight to reflect partial success:

\[
\text{Win Rate} = \frac{\text{Wins} + 0.5 \times \text{Settlements}}{\text{Total Cases}}
\]

This metric provides a coarse signal for whether MARL training is converging toward competent adversarial behaviour. It is interpreted alongside the two quality metrics below, since a high win rate achieved through procedurally invalid arguments is not considered acceptable performance.

\subsubsection{Simulator: Procedural Compliance}

Procedural compliance measures how consistently agents operate within valid Thai civil procedure. An action is classified as valid if it is permissible at the current stage of the trial under the Civil Procedure Code:

\[
\text{Procedural Compliance} = \frac{\text{Valid Actions}}{\text{Total Actions}}
\]

Low compliance indicates that agents are exploiting simulation gaps rather than learning transferable courtroom behaviour, which directly undermines the training value of the platform.

\subsubsection{Simulator: Hallucination Rate}

Legal reasoning quality is assessed by auditing the statutory and precedent citations made by each agent. A claim is flagged as a hallucination when it cannot be traced to an extant provision of the Thai Civil and Commercial Code or to a citable court decision:

\[
\text{Hallucination Rate} = \frac{\text{Unverifiable Legal Claims}}{\text{Total Legal Claims}}
\]

Reducing hallucination rate is a prerequisite for the platform to serve as a meaningful legal training tool. Unverifiable citations would mislead users and erode the simulation's value as a proxy for real proceedings.
